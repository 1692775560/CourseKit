# CourseKit 课程报名管理器

CourseKit 是一个为 **vibe coding / gstack 实战课** 准备的最小真实项目。它不是玩具页面，而是一个可以用于课堂演示完整 AI 编程流程的小型前端应用：从需求澄清、计划评审、实现、代码 review、浏览器 QA，到最终 ship。

项目主题：

> 管理员在课程列表页查看课程名称、负责人、报名人数、容量和状态，并按状态筛选课程、统计当前结果、导出当前筛选结果为 CSV。

## 这个项目用来教什么

这个仓库适合用来讲：

- 什么是 vibe coding
- 如何把模糊需求变成可执行任务
- 如何用 gstack 做需求澄清和计划评审
- 如何让 AI 写代码但不盲信 AI
- 如何补测试、做 review、做浏览器 QA
- 如何把一次 AI 编程变成可交付的 PR

推荐课堂主线：

```text
/office-hours
  -> /plan-ceo-review
  -> /plan-eng-review
  -> /plan-design-review
  -> 实现
  -> /review
  -> /qa
  -> /ship
```

## 项目功能

当前版本已经实现：

- 课程列表页
- 状态筛选：全部、招生中、已满、已关闭
- 当前筛选结果数量，例如 `显示 2 / 6 门课程`
- 导出当前筛选结果为 CSV
- CSV 字段转义：逗号、双引号、换行
- 空结果状态
- 无结果时禁用导出按钮
- 桌面端表格布局
- 移动端卡片式布局，避免表格挤压
- Node 内置单元测试

## 技术栈

这个项目刻意保持简单，方便课堂使用：

- HTML
- CSS
- 原生 JavaScript ES Modules
- Node.js 内置 HTTP server
- Node.js 内置 test runner
- 无 React/Vue/Vite 依赖
- 无安装步骤

这样设计的原因是：课堂重点是 AI 编程流程，而不是前端框架配置。

## 课堂怎么用

### 用法一：作为最终答案项目

教师可以直接展示当前仓库，让学生看到一个完整交付应该包含什么：

- 功能可运行
- 逻辑拆到了 `courseUtils.js`
- CSV 边界有测试
- 移动端布局可用
- README 说明清楚

适合课程前 10 分钟快速展示“最后我们要做到什么程度”。

### 用法二：作为学生实操项目

教师可以先创建一个教学分支，把功能退回到基础列表页，只保留课程数据和表格，然后让学生使用 gstack 实现功能。

建议需求如下：

```text
当前项目有一个课程列表页。管理员可以看到课程名称、报名人数、容量和状态。
现在要新增一个“筛选和导出”功能：

1. 按课程状态筛选：全部、招生中、已满、已关闭。
2. 显示当前筛选结果数量。
3. 支持导出当前筛选结果为 CSV。
4. 移动端不应挤压表格内容。
5. 至少补一组单元测试或组件测试。
```

学生按这个流程执行：

```text
/office-hours
/plan-ceo-review
/plan-eng-review
/plan-design-review
实现代码
/review
/qa http://127.0.0.1:5173
/ship
```

### 用法三：作为 review / QA 训练项目

教师可以故意制造几个 bug，让学生用 `/review` 和 `/qa` 找出来。

可制造的 bug：

- CSV 导出使用全部课程，而不是当前筛选结果。
- CSV 没有处理双引号和换行。
- 筛选结果为空时导出按钮仍然可点击。
- 移动端表格列被挤压。
- 测试只检查函数存在，不检查真实行为。

学生任务：

```text
/review

请 review 当前分支，重点检查：
1. CSV 导出是否只导出当前筛选结果。
2. CSV 转义是否正确。
3. 空结果和按钮状态是否一致。
4. 移动端是否可用。
5. 测试是否覆盖关键路径。
```

然后运行：

```text
/qa http://127.0.0.1:5173
```

## 推荐 gstack 演示脚本

### 1. 先让 AI 读项目

```text
请先阅读这个项目，不要改代码。总结技术栈、启动命令、测试命令、关键页面、现有代码约定。
```

### 2. 需求澄清

```text
/office-hours

我们要在 CourseKit 课程列表页增加状态筛选和导出当前筛选结果为 CSV 的能力。用户是课程运营同学，他们每天要查看哪些课还在招生、哪些课已满，并把当前列表发给团队。请先帮我把需求想清楚，不要写代码。
```

期望产物：

- 用户是谁
- 真实痛点是什么
- 最小可交付版本是什么
- 本次不做什么
- 未来可以扩展什么

### 3. 范围评审

```text
/plan-ceo-review

基于刚才的设计文档做范围评审。请给出本次必做、可选增强、明确不做，并说明为什么。
```

建议课堂上选择：

```text
Selective Expansion
```

也就是保留小范围，但加入少数高价值增强，例如：

- CSV 文件名包含日期
- 无结果时禁用导出按钮

### 4. 工程评审

```text
/plan-eng-review

基于最终范围做工程评审。请输出文件级计划、数据流、边界条件、测试计划，以及给 /qa 使用的重点测试路径。
```

期望学生看到：

- 应该修改哪些文件
- 数据流是什么
- CSV 有哪些边界条件
- 哪些路径必须有测试
- QA 应该如何验证

### 5. 设计评审

```text
/plan-design-review

请检查筛选器、结果数量、导出按钮、空状态和移动端布局。只评审计划，不写代码。
```

### 6. 实现

```text
请按照已经通过的 office-hours、CEO review、eng review 和 design review 实现功能。

要求：
1. 严格遵守 not in scope。
2. 不引入新依赖，除非先说明理由。
3. 写实现代码时同步补测试。
4. 完成后运行项目测试。
5. 汇报修改文件、测试结果、未完成事项。
```

### 7. 代码 review

```text
/review

请 review 当前分支相对 main 的 diff。重点检查：
1. 数据正确性。
2. CSV 导出是否只导出当前筛选结果。
3. CSV 转义是否正确。
4. 空状态和按钮状态。
5. 测试是否覆盖关键路径。
```

### 8. 浏览器 QA

先启动：

```bash
npm run dev
```

再运行：

```text
/qa http://127.0.0.1:5173 --quick

重点测试：
1. 默认显示全部课程。
2. 切换每个状态筛选。
3. 当前结果数量正确。
4. 导出按钮在有结果和无结果时状态正确。
5. 导出内容只包含当前筛选结果。
6. 移动端 375px 宽度不重叠。
```

### 9. Ship

真实推送：

```text
/ship
```

课堂只演示检查流程时：

```text
请模拟 /ship 的检查流程，不要 push，不要创建 PR。输出真实运行前还缺什么。
```

## 学生验收标准

学生完成练习后，应提交：

- `/office-hours` 输出摘要
- `/plan-ceo-review` 的范围决策
- `/plan-eng-review` 的测试计划
- 最终代码 diff 或 PR
- 测试输出
- `/review` 发现的问题和处理记录
- `/qa` 报告或截图
- 自评：如果不用 gstack，这次最可能漏掉什么？

评分建议：

| 维度 | 满分 | 观察点 |
|---|---:|---|
| 需求澄清 | 20 | 是否有用户、场景、范围、not in scope |
| 工程计划 | 20 | 是否有文件计划、数据流、边界条件、测试计划 |
| 实现质量 | 20 | 是否遵守项目约定、无不必要依赖 |
| 测试质量 | 15 | 是否覆盖筛选、CSV、空状态、回归路径 |
| Review 闭环 | 10 | 是否处理 `/review` 发现的问题 |
| QA 闭环 | 10 | 是否真实浏览器验证并保留证据 |
| 发布准备 | 5 | PR 描述、测试结果、风险说明是否清楚 |

## 给学生的提示

不要一上来就说：

```text
帮我写这个功能。
```

更好的做法是：

```text
请先读项目，不要改代码。
```

然后按顺序推进：

```text
想清楚需求 -> 评审范围 -> 评审工程方案 -> 实现 -> review -> QA -> ship
```

AI 写代码很快，但快不是交付。交付需要判断、测试、review 和真实运行。
